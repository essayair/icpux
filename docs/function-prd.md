# 一、主数据管理

__模块定位__

主数据管理是全平台共享的基础档案中心，统一管理往来单位、用户权限、物料、仓库库位、车辆、价格等基础资料，保证各模块数据口径一致。主数据为采销、物流、库存、质检、地磅、监控等所有业务模块提供统一的引用来源，同时完成权限管理与组织架构配置。

基础智能化工具

OCR识别等AI基础能力

__关键交互__

• 各业务模块下拉/引用主数据，保证数据来源唯一，避免重复录入。

• 主数据变更时，系统记录版本历史；已生效单据不受主数据变更影响，新单据引用最新版本。

• 权限配置决定用户可见的业务线与操作范围，尤其在合作伙伴共管场景下实现数据隔离。

• 未来引入第三方信用管理、风控模块，作为底层应用功能扩展。

__与其他模块的关联__

主数据是所有业务模块的基础依赖。采销合同引用供应商/客户/物料/价格主数据；物流模块引用车辆主数据；库存模块引用仓库库位主数据；组织架构为所有模块提供部门组维度的权限与统计口径。

# 二、合同管理（采销系统）

__模块定位__

采销系统是平台的业务起点，管理采购与销售的合同、订单全流程，自动拆分双边业务，管控额度、价格与结算，沉淀供应商与客户主数据。下游物流、库存、质检、结算的业务均由这里的订单驱动。

从贸易业务角度来看，合同管理覆盖业务合同的发起、执行和后续的状态追踪，合作对象是上游的供应商和下游的客户。合同分为采购合同、销售合同以及双边合同，共用一套模板体系。同时支持框架合同/协议（年度协议、月度协议），协议下可拆分生成单次合同，带出基本信息后重新进入审批流程。

__关键业务流程__

① 采销员录入合同 → 判断双边/单边 → 双边自动拆出采购单与销售单。

② 下单时系统校验额度，超额弹窗提示或拦截（一期人工控制，可做合作伙伴总额提示）。

③ 合同生效后 → 由合同下达供应商发货指令/提货指令 → 自动通知物流与库存。

④ 收到收货确认 → 结算引擎按规则算出预付款/尾款 → 生成结算单待财务审核。

⑤ 合同执行过程中持续跟踪：发货指令在途状态 → 物流入库 → 业务入库 → 结算 → 发票 → 付款。

## 3、双边合同

__业务说明__

双边合同管理"以销定购"业务模式。在同一合同内同时约定上游采购和下游销售，系统自动拆分为采购单与销售单，实现采销两端的协同管理。

核心流程：签订双边合同 → 自动拆分采购单\+销售单 → 采购端生成供应商发货指令（供应商提货） → 销售端生成销售提单 → 以销定购，__按销售需求驱动采购执行__。

# 三、物流管理

__2、物流入库单__

货物已过磅进入仓库。由发货指令、运输单据、地磅数据进行确认。操作流程：选择发货指令 → 选择物流运单 → 对物流运单进行入库确认。或单独选择物流运单进行入库。

关联关系：物流入库单 → 业务入库单（库存形成） → 采购合同（履约跟踪）。

__3、物流出库单__

覆盖常规销售出库和以销定采直拨两种核心场景：

• 常规销售出库：销售提单 → 物流出库单 → 过磅出库 → 库存扣减。

• 以销定采直拨：供应商直接发到下游客户（直拨库），基于物流出库单自动反向生成物流入库单。下游确认收货后，关联上游合同完成入库，再完成出库。

• 退货场景：支持销售退货和采购退货的逆向物流流程。

__关键交互（用户怎么用）__

① 采销下达发货/提货指令 → 物流生成运输任务 → 叫车派车。

② 车辆进入厂区电子围栏 → 自动推送"待收货预警"给现场人员。

③ 在途异常（超时/偏航/停留）→ 自动告警并挂起相关单据。

④ 运输完成 → 生成台账 → 按运价核算运费 → 与承运商对账结算。

__与其他模块的关联__

物流管理上承采销系统的发货指令/提单，下接库存管理的入库/出库单，横向对接第三方物流平台获取实时状态，并为财务结算提供运费数据。地磅过磅数据是物流入库/出库的数量确认依据。

__1、业务入库单__

引用物流入库单创建业务入库，完成收货确认，确认货权及库存形成。由物流入库单、采购合同、质检结果驱动。__财务确认并过账后__，库存正式形成且批次成本开始计算。

关键规则：

• 业务入库单未过账时（业务状态"业务已确认"、入账状态"未生成"），可以做销售提单（预出库）。

• 财务"确认"和"过账"合并为一个按钮"确认并过账"。

• 支持无采购入库场景（创建新批次）、商品编码转化、定价调整单等特殊业务。

__2、销售出库（销售提单）__

销售提单/发货通知单是销售出库的核心单据，向下关联物流运输单和物流出库单。支持以下出库模式：

• 常规出库：按销售合同创建提单 → 拣配库存 → 物流出库 → 过磅确认。

• 直拨出库：供应商直接发货给客户，不经过我方仓库，但系统需补完入库\+出库流程。

• 销售退货：支持退货到原批次或创建新批次入库。

__关键交互（用户怎么用）__

① 物流到货 \+ 过磅 \+ 质检合格 → 生成入库单与批次 → 库存形成。

② 销售指令 → 出库单（销售提单）→ 拣配 → 物流出库 → 扣减库存。

③ 移库登记源/目标库位 → 更新库存位置。（销售移库场景：内部利润中心按移库核算）

④ 盘点录入实盘量 → 系统比对账面 → 生成差异单 → 报损/报溢处理。

__与其他模块的关联__

库存管理上承物流管理（入库/出库单），下接财务结算（成本核算），横向对接质检化验（合格方可入库）、风控中心（敞口库存计算）、地磅管理（数量确认）。批次成本数据同时为资金计息和财务报表提供基础。

__实现方案建议__

工作台建议采用后端逻辑实现（而非Agent），原因如下：

• 待办事项需要实时、准确地聚合多模块数据，后端SQL/API查询更可靠。

• 业务概览涉及数据统计和图表渲染，后端计算\+前端可视化是最成熟的方案。

• 预警通知需要定时任务和消息推送机制，属于系统基础能力。

• Agent可以作为辅助层，提供自然语言查询和智能建议（如"本周合同执行异常汇总"），但不适合作为核心实现。

• 合作伙伴门户与客户端：Web后台 \+ 小程序/企业微信 \+ API对接

• 信息同步与主动推送：按相关方推送合同、物流、磅单、仓储状态、监控影像

• 按主体隔离 \+ 按关系授权：单据从生成即绑定签约企业主体，合作伙伴只看到与其相关的单据

• 叠加物联网客观数据（地磅、监控），用"机制\+数据"而非"关系"建立信任

__B\. 监管仓标准化服务__

定位：监管仓是最"绕不开"、也最能体现核心价值的服务——靠控货（物联网监管\+可信单据\+现场监控）为合作伙伴与资金方提供可信的货权与货物状态，把"关系型信任"替换为"机制型信任"。

包含能力：入出库核验、电子仓单/监管凭证；实时货物状态监控（地磅、电子围栏、视频、传感）；货权登记与权属（留置/动产抵押）管理；面向资金方的监管报告与穿透式查询。

__C\. 供应链金融服务__

定位：基于监管仓的可信货权与确权应收账款，对接资金方（物产/银行/保理），为合作伙伴提供融资，自身也降低融资门槛与成本。

包含能力：确权数据包组装并推送资金方（磅单\+质检\+确权\+监管凭证）；融资申请、放款进度追踪、资金路由（多资金方切换）；基于一期可信交易数据的信用/授信与买方信用评估。

__D\. 财务结算与资金中心__

定位：在一期采销/物流结算之上，管理合同收付款、账期、利息与对账，支撑多主体资金归集与内部结算。

包含能力：合同收款/付款、账期与利息管理；多主体/利润中心资金归集与内部结算；对账、资金看板与资金链预警。

__E\. 区域产业数据与产业研究能力__

定位：采集、掌握区域产业数据，目的是形成我们自己的判断力与运营能力（核价、风控、找货、议价）。资讯是产业研究能力，不是对外售卖的服务。

包含能力：数据采集（周边矿山/选厂的储量、产能、库存、运输距离、价格、成本）；一线采集工具（轻量录入/上报工具）；数据沉淀（通过产业关系获得数据、换数据并结构化入库）；产业研究产出（区域产业地图、价格基准/指数、供需热力）。

__F\. 可信数据层__

定位：为监管与金融提供"不可篡改"的数据底座，把"过磅—质检—确权—单据"链路用存证/直连固化。

包含能力：区块链/可信时间戳存证、加密；关键数据设备直连（直采，减少人工录入空间）；监控影像切片上云、关键节点上链。

__G\. 多企业主体/利润中心管理__

定位：支撑"用不同企业主体跟合作伙伴签合同"的现实，在一个系统窗口内管理多主体，并从源头解决数据隔离与共享。

包含能力：多主体/利润中心/业务组在单一系统窗口管理；单据从生成即绑定签约主体，按主体隔离、按关系授权；内部利润中心间核算（如销售移库）。

__H\. 模型/能力升级__

定位：把一期沉淀的数据进一步转化为更强的能力；审核从人工流程逐步走向半自动/自动，体现"分阶段、不贪全"。

包含能力：审核模型（人工流程→半自动/自动）；质量异常检测预警、损耗评估、成本核价模型深化；买方/合作方信用画像（可对资金方输出，降低风控成本）。

# 十一、其他关键功能与特性

__单据图像识别（OCR）__

支持磅单、发票、合同、提单等单据的OCR识别与结构化提取。相关业务流程完结需要完整相关附件，作为必选项。

__双轨并行过渡__

系统上线后，考虑到很多业务流转会在微信或其他工作群确认，短期内无法实现全流程数字化。系统需支持线下操作结果的快速补录，确保系统数据与实际业务保持一致。设置过渡期后逐步收紧流程管控。

__移动端支持__

二期需覆盖小程序/企业微信移动端，支持业务订单录入、审批处理、物流跟踪、过磅拍照等移动场景。

__开发周期与资源参考__

• 传统模式：大宗贸易运营管理系统，第三方ERP基础版SAAS部署20\-50万/年，常规SAP年费百万，且扩展支持成本高；从0到1成本百万起步，开发周期3\-6个月。

• AI支持模式：文档确定后1\-2月可进入测试阶段，3个月线上测试。后续功能扩展可在需求/解决方案确定后数天内完成测试并上线。

• 资源需求：AI高级版本 \+ 前期外包技术咨询。
