## 一、项目概述

### 1.1 项目名称与命名决策

候选：`jiayi86`、`jiayiscp`、`jiayiicp`、`RICSE/RICE`

建议把"三个名字"分开，避免混用造成沟通成本：

| 用途 | 建议取值 | 理由 |
|------|---------|------|
| 产品/生态品牌名 | **RICE（Regional Industry Chain Ecosystem）** | 直接表达"区域产业链服务生态"的产品愿景，对外可讲故事；RICSE 略长，RICE 更顺口 |
| 内部代号/仓库名 | **jiayiscp** | 含主体 jiayi + 业务内核 SCP，工程上语义清晰；jiayi86 偏中性、可作组织级总域名 |
| 主域名 | jiayi86.com / jiayiscp.com（择一） | ⚠️**待确认**：域名可注册性需实查，本文档不预设可用 |

> ⚠️**待确认**：以上为建议方案，最终命名由评审拍板。命名一旦确定写入 `Sot.md`，代码仓库、包名、数据库名统一沿用，不再二次更名。

### 1.2 项目背景与定位

本系统服务于**区域产业链的供应链协同规划**，分三个递进层次：

1. **SCP 供应链规划**（Supply Chain Planning）：需求计划（销量预测）、供应计划（采购备料）、生产计划（排产排程）、库存计划（安全库存）、配送计划（物流网络优化）。
2. **集成供应链/产业链规划**（Integrated/Industry Chain Planning）：将上述模块端到端打通，做一体化协同，而非各自为政。
3. **区域产业链服务生态**（RICE）：在一体化规划之上，构建面向区域多主体的产业链服务平台。

> 定位说明：这是一个**重型 B 端、领域复杂度高**的系统，核心价值在业务链的精确编排，而非"功能大而全"。这一判断直接决定了后面的架构取舍（见第二章）。

### 1.3 项目目标

**业务目标（一期）**
- 走通**合约主线一条完整路径**：单一角色（采购/交易员）下，合约 `创建 → 审核 → 生效`。
- 验证产品设计、数据模型、技术架构、AI 开发流程四者是否能闭环跑通。

**工程目标（贯穿全程）**

| # | 目标 | 衡量标准 |
|---|------|---------|
| 目标 1 | **项目 AI OS 建设** | 产品文档/开发技术/流程完整、可记录、可追溯、可优化成长 |
| 目标 2 | **AI 开发更准、成本可控、人机均可维护** | AI 产出准确率提升、Token/工时成本可度量、人工可独立调试任意模块 |

## 二、技术方案

### 2.1 技术路线总原则

> **不依赖一体化业务脚手架（不套用芋道这类全家桶），但使用主流前后端框架；以 AI 友好为第一优先，同时保证人工可调试、可优化。**

四条铁律：
1. **不套脚手架，但不重复造轮子**——基础能力（认证、权限、审批、文件、消息）优先选 Node 生态成熟库拼装，而非从零手写，也不引入芋道这类强耦合一体化框架。
2. **全栈 TypeScript**——前后端类型贯通，让 AI 一次理解全栈数据流（这是不选 Java 系脚手架的核心理由）。
3. **领域驱动（DDD）**——用充血模型 + 有界上下文表达复杂业务编排，避免贫血事务脚本随时间腐化。
4. **AI 产出、人工确认**——架构与代码由 AI 主导产出，传统 B 端技术顾问做关键评审与兜底（尤其重型场景）。

### 2.2 技术栈选型（锁定）

| 层次 | 选型 | 说明 |
|------|------|------|
| 后端框架 | **NestJS + TypeScript** | 模块化、DI/Guards/Interceptors，企业级模式 |
| 架构模式 | **模块化单体 + DDD 有界上下文** | 单进程、强一致事务、预留未来拆分能力 |
| ORM | **Prisma** | Schema-first、类型生成、自动迁移、Studio 可视化 |
| 主数据库 | **PostgreSQL 15+** | 复杂关联查询与事务完整性强，适配规划/结算场景 |
| 缓存/队列 | **Redis + BullMQ** | 缓存、会话、后台任务（审批超时、定时报表） |
| 文件存储 | **MinIO（自建 S3）/ 阿里云 OSS** | 附件、合同扫描件 |
| 实时通信 | **Socket.IO + Redis Adapter** | 工作台通知、待办推送 |
| 前端框架 | **Next.js 14+（App Router）** | RSC 默认、Nested Layouts 匹配后台系统 |
| UI 库 | **Ant Design 5** | ProTable/ProForm/ProLayout 提效 |
| 服务端状态 | **TanStack Query** | 缓存/失效/乐观更新 |
| API 规范 | **REST + OpenAPI（@nestjs/swagger）** | 自动生成文档，接口量大时维护可靠 |

> ⚠️**待确认（信创风险）**：若未来面向国企/政府客户，可能被要求兼容国产库（达梦 DM）。**Prisma 对达梦支持有限**，需在一期就确认是否预留抽象层或评估替代 ORM。这是本架构最需要顾问把关的一点。

### 2.3 代码组织、架构模式与一期范围

**代码组织：Monorepo（单体仓库）** ✅ 已确认

前端（Next.js）与后端（NestJS）是两套独立程序，但统一放在**一个 git 仓库**内管理，采用 **pnpm workspaces**。前后端**独立部署**（仓库统一 ≠ 部署统一）。

选 Monorepo 的两条核心理由，均直接对应本项目战略：
1. **全栈 TS 类型共享**——前端直接 import 后端定义的类型，改一处两端同步；分仓需发 npm 包传类型，该优势基本作废。
2. **AI 友好（关键）**——AI 在一个目录内同时看到前端、后端、数据库 Schema、文档的完整上下文；分仓会物理切断上下文，与"选模块化单体而非微服务"拒绝的"上下文分散"是同一个坑。

> 工具：一期用 **pnpm workspaces** 起步（轻、AI 易理解），项目变大再上 Turborepo 加构建缓存；Nx 一期过重，不用。

**完整目录布局：**

```
jiayiscp/                       ← 一个 git 仓库（Monorepo）
├── apps/
│   ├── web/                    ← 前端 Next.js（独立部署）
│   └── api/                    ← 后端 NestJS（独立部署，模块化单体）
│       └── modules/
│           ├── contract/          ✅ 一期实现（合约：创建→审核→生效）
│           ├── master-data/       ⬜ 仅建合约必需的最小主数据
│           ├── common/            ✅ 一期实现最简版（认证、RBAC、审批）
│           ├── logistics/         ⛔ 本期不铺开
│           ├── inventory/         ⛔ 本期不铺开
│           ├── quality-inspection/⛔ 本期不铺开
│           ├── weighbridge/       ⛔ 本期不铺开
│           ├── finance/           ⛔ 本期不铺开
│           └── workbench/         ⬜ 承载合约待办的最小工作台
├── packages/
│   └── shared-types/           ← ⭐ 前后端共享 TypeScript 类型
├── prisma/
│   └── schema.prisma           ← 数据库结构定义（结构真相所在）
├── docs/                       ← Sot.md / Claude.md / 开发规则 / prompt记录
└── package.json
```

**架构模式：模块化单体 + DDD。** 后端为单进程，按 DDD 有界上下文划分模块；**一期严格收口到合约主线**，其余上下文先留空目录、不铺开实现。每模块内部分层：`domain / application / infrastructure / interfaces`。模块间通信：同步走应用服务调用，异步走领域事件总线；**严禁跨模块直接访问对方 Repository 或表**。

### 2.4 自建能力清单（"方向A但不造轮子"）

⚠️ 不套脚手架意味着 RBAC、审批流需自建。**一期做最简版，后续迭代细化**。落地方式如下：

| 能力 | 一期做法（最简版） | 推荐选型 | 后续迭代 |
|------|-----------------|---------|---------|
| 认证 | 账号密码 + JWT | Passport（NestJS 官方集成） | SSO、多端、三方登录 |
| 授权 RBAC | 角色-权限硬编码够用，先支持"采购/交易员"单角色 | Casbin（成熟策略引擎） | 数据权限、按钮级、多租户 |
| 审批流 | **不自研引擎**，先做"合约审核"单一固定流程 | 先固定流程；评估挂 Flowable REST 或 Node 工作流库 | 可视化流程设计、会签/或签 |
| 文件 | 直接接 MinIO/OSS SDK | MinIO（自建）/ OSS | 影像关联、版本管理 |
| 消息/异步 | BullMQ 跑审批超时提醒 | BullMQ | 复杂编排、跨模块联动 |

> 关键原则：**审批引擎是最容易低估工作量的坑**。一期绝不自研通用 BPM，只实现"合约审核"这一条写死的流程；通用化挪到后续迭代再评估成熟引擎。

### 2.5 为何不选芋道（一句话归档）

芋道（Java + Spring + MyBatis）能开箱白嫖 RBAC/多租户/工作流/代码生成，起步快；但它是 Java 全栈（与本项目"全栈 TS + AI 友好"战略相悖）、贫血事务脚本模型（不利于产业链复杂编排）。本项目核心价值在领域建模，故选自建路线，代价是基础设施需自拼——已在 2.4 用成熟库把该代价压到最低。

### 2.6 数据库管理边界（开发约定） ✅ 已确认

数据库（PostgreSQL）是**与后端平级的独立服务**，不是后端的子集。"管理"分两层，职责清晰：

| 事项 | 归属 | 说明 |
|------|------|------|
| 表/字段结构定义 | **代码（Prisma）** | `schema.prisma` 是结构的唯一真相，进 git |
| 结构变更（建表/改字段） | **代码迁移** | 一律走 `prisma migrate`，生成 DDL 交库执行 |
| 数据增删改查 | **代码（Prisma Client）** | 收口到各模块 Repository 层 |
| 事务、并发、存储 | **数据库引擎** | PostgreSQL 自身负责 |
| 实例启停、备份、扩容 | **运维/部署层** | 属预算「基础设施/云」项，非代码职责 |

**两条铁律（写入 `开发规则.md` 强制执行）：**
1. **禁止手动 DDL**——生产环境严禁人手连库改表/改结构。结构真相在 `schema.prisma`，手改会导致代码与库不一致、下次迁移冲突。结构变更只走 Prisma migrate。
2. **数据访问收口 Repository**——禁止跨模块直接访问对方表，所有库操作经本模块 Repository 层，与 DDD 跨上下文隔离一致。

> 开发/测试阶段可用 Prisma Studio 可视化查看数据；但**结构变更永远走代码迁移**，不在 Studio 或客户端手改表结构。

## 三、AI OS 建设（Context → Connections → Capabilities → Cadence）

用 **4C 框架**管理、调度、运行 AI 开发流程。每个 C 对应一组可记录、可追溯的制品：

### 3.1 4C 框架总览

| 维度 | 含义 | 核心制品 | 解决什么 |
|------|------|---------|---------|
| **Context** 上下文 | AI 知道什么 | `Sot.md`（信息源）、产品文档、架构文档、`Claude.md` | 让 AI 拥有完整、最新、单一可信的项目知识 |
| **Connections** 连接 | AI 能触达什么 | 代码仓库、Prisma Schema、OpenAPI 文档、MCP 连接器 | 让 AI 直接读到真实代码/库结构，而非凭记忆 |
| **Capabilities** 能力 | AI 能做什么 | 开发规则、代码规范、可复用 Prompt 库、Skill/Agent 配置 | 约束 AI 产出风格，沉淀可复用能力 |
| **Cadence** 节奏 | 工作如何流转 | 迭代节奏、评审门禁、更新记录、Prompt 记录 | 让 AI 工作可调度、可回溯、可优化成长 |

### 3.2 文档与制品清单（落地目标1）

```
/docs (纳入 git，可追溯)
├── Sot.md            # 信息源/Source of Truth：项目所有决策的单一真相，本文档定稿后纳入
├── Claude.md         # AI 协作约定：项目背景、技术栈、目录约定、禁忌、常用指令
├── 开发规则.md        # 编码规范、DDD 分层约定、命名、提交规范
├── 产品文档/          # 持续更新，每次变更追加"更新记录"
│   └── 更新记录.md     # 谁、何时、改了什么、为什么
└── prompt记录/        # AI prompt 归档：有效 prompt 沉淀为可复用模板
```

**可记录 / 可追溯 / 可优化成长**的实现：
- **可记录**：所有决策进 `Sot.md`，所有文档变更进"更新记录"，所有有效 prompt 进 `prompt记录`。
- **可追溯**：git 版本 + 文档间交叉引用，任一决策可回溯到来源与时间。
- **可优化成长**：定期回看 prompt 记录与产出质量，沉淀模板、修订 `Claude.md` 规则。

### 3.3 三个工程目标的落地映射

| 目标 | 由哪个 C 承载 | 落地动作 |
|------|------------|---------|
| 目标1 文档完整可追溯 | Context + Cadence | 建立上述 `/docs` 体系并强制更新纪律 |
| 目标2 AI准、成本可控、人机可维护 | Connections + Capabilities | AI 直读 Prisma/OpenAPI 减少臆测；开发规则约束产出；REST+Swagger 保证人工可独立调试 |
| 目标3 技术架构可运行 | 全部4C | AI 产出架构 → 顾问确认 → 前后端/库/服务器跑通 |

| 域名+证书 | 约 60–200/年 | 一次性/年付 |
| AI 编码工具 | 按订阅/用量 ⚠️待核 | 建议先按团队人头估订阅，用量上线后再优化 |

> 结论：**一期预算 ≈ 人力成本 + 少量基础设施（千元/月级）+ AI 工具订阅**，基础设施不是大头。填入真实人力数即可得出总额。
